# M6 Patch Transaction Design

## Status And Boundary

- Design status: implemented as a server-side transaction core and verified in synthetic temporary workspaces.
- Production API/UI wiring: implemented behind default-off `apply_patch`, proposal-bound approval and preflight gates.
- Real workspace proposal application: not authorized and not executed.
- Git, package/config changes, dependency install and remote actions: excluded.

## Atomicity Definition

General file systems do not provide one atomic rename covering multiple files. This design therefore uses a bounded transaction contract instead of claiming simultaneous multi-file replacement:

1. Every preimage is read and SHA-256 verified before the first source mutation.
2. Every postimage is generated in memory from strict unified-diff hunks and must match its declared SHA-256.
3. Every postimage is written and fsynced before the commit phase begins.
4. Each original is renamed to a transaction backup before its prepared postimage is renamed into place.
5. Any process-visible failure before the durable `committed` journal state triggers reverse-order rollback.
6. On restart, `prepared`, `committing` and `rollback_failed` transactions recover toward all-preimage state; `committed` transactions preserve postimages and finish cleanup.

The durable `committed` journal state is the transaction commit point. Before that point, recovery chooses rollback. After that point, recovery chooses completion.

## Transaction Layout

Synthetic and future approved executions use a fixed workspace-local root:

```text
<workspace>/ai-output/.agenthub-patch-transactions/<transaction-id>/
  journal.json
  00.next
  00.backup
  01.next
  01.backup
```

- `*.next`: fsynced postimage prepared before commit.
- `*.backup`: original file moved during commit and retained until the durable commit point.
- `journal.json`: proposal identity, proposal SHA-256, state and per-file before/after hashes. It contains no source or patch body.
- Unknown files inside a transaction directory block automatic cleanup.

## State Model

| State | Meaning | Recovery decision |
|---|---|---|
| `prepared` | All postimages and the journal are durable; no commit promised | Restore all preimages |
| `committing` | One or more per-file renames may have occurred | Restore all preimages |
| `committed` | Every target matches its after SHA-256 and the commit point is durable | Preserve postimages and finish cleanup |
| `rollback_failed` | Automatic rollback found an unexpected/missing artifact | Stop, preserve evidence and require manual review |

## Deterministic Postimage Rules

- Input must already pass the M6 proposal validator.
- Source must be an existing regular UTF-8 file with no symlink/junction path component.
- `beforeSha256` must match the exact source bytes.
- Unified-diff path headers, hunk counts, hunk positions, context lines and removal lines must match exactly.
- LF and CRLF are supported; mixed line endings are rejected.
- Source reads are limited to 256KB per file; generated postimages are limited to 512KB per file.
- `afterSha256` must match the exact generated bytes before any source mutation.

## Failure And Recovery Matrix

| Observed artifacts | Interpretation | Action |
|---|---|---|
| target=before, next exists, no backup | File not committed | Delete `next`; keep target |
| target missing, backup=before, next exists | Original moved, replacement not installed | Restore backup; delete `next` |
| target=after, backup=before | Replacement installed before commit point | Delete target; restore backup |
| target=before, backup=before | Original already restored | Delete redundant backup |
| target has unknown hash | Possible external modification | Refuse overwrite; preserve transaction evidence |
| target and backup both missing | Preimage cannot be proven recoverable | Mark recovery failed; preserve evidence |

## Production Wiring Status

Production-local wiring now enforces all of the following:

1. Exact `proposalId`, proposal SHA-256 and every target path are carried by the registered proposal.
2. Application requires a successful proposal-specific preflight record for the same workspace and run.
3. `apply_patch` is false for all eight Agents by default and application requires a separately clicked single-use ticket.
4. Startup, workspace switching and pre-application recovery block on any unresolved transaction; switching also waits for active model calls and controlled actions.
5. Every preimage is rechecked immediately before transaction preparation and commit.
6. Runtime and browser responses contain summary paths/hashes only, never source or raw patch bodies.

Real application remains independently gated: the live service must first hold a concrete proposal, and the user must approve that runtime proposal ID, proposal SHA-256 and exact paths. Production wiring approval alone does not authorize real application.

## Remaining Non-goals

- No real proposal submission, real preimage read or real source mutation.
- No Git staging, commit, rollback or push.
- No binary patch, rename, file creation/deletion or dependency/config modification.
