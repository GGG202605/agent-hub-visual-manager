# AgentHub Coordination Contract v1.0

日期：2099-01-01

## 1. 唯一口径

运行时唯一契约由 `src/lib/coordinationContract.ts` 提供。旧版 mock、Wiki v0.1/v0.2 和导入数据可以作为来源，但必须先映射到本契约，不得把多套状态字符串直接混入运行引擎。

8 个规范角色：

1. `AG-COORD`：确定性监督者，接收需求、拆解、分派和升级，不执行副作用。
2. `PRO`：专业评审，只给建议，不能代替用户授权。
3. `UI-PRODUCT`：UI/产品体验专家，按需激活。
4. `AG-DEV`：在批准路径内实现并提供测试证据。
5. `EXECUTOR`：确定性工具服务，不是自由对话 LLM。
6. `AG-SEC`：权限与安全门禁，缺少证据时不宣告通过。
7. `AG-REVIEW`：独立质量复核，不能复核自己的产物。
8. `HANDOFF`：交接和恢复快照，不继承权限。

## 2. 完成定义

真实 `completed` 必须同时满足：

- 所有激活任务均为 `accepted`；
- 至少存在一个真实产物且哈希验证通过；
- 所需人工审批已经满足；
- AG-SEC 与 AG-REVIEW 的 High/Medium 发现为 0；
- 每个真实 Agent 阶段均有成功结果，失败和阻塞不得被动画计时器覆盖。

模板话术运行只能进入 `simulated`，不得显示为真实闭环完成。

## 3. M2 执行安全

当前真实动作仍只允许：

- `save-note`：写入审批时工作区的 `ai-output/`；
- `run-build`：在审批时工作区运行固定 `npm run build`。

每次动作必须包含：

- `runId`；
- `agentId`；
- `idempotencyKey`；
- 当前工作区 ID；
- 动作参数 SHA-256；
- 60 秒有效、单次使用的审批 ID 与 HMAC 票据。

执行期间即使 UI 切换工作区，动作和回执仍绑定原工作区；同一 run 不允许并发副作用。回执按工作区分区，并通过 `previousHash + receiptHash` 形成可核验链。

## 4. 当前边界

- M2 的审批票据由用户点击现有明确动作按钮触发，解决直接调用 `/api/execute` 绕过 UI 语义的问题。
- 本地会话令牌用于防止无会话、过期页面和非允许浏览器来源的误调用；它不声称能对抗已取得当前操作系统用户权限的恶意本机进程。
- M3 仍需增加逐 Agent 权限矩阵、待审批队列、拒绝/撤销和持久化策略。
- 本阶段尚未实现任务 DAG、代码 diff 应用、自动修复、Git 自动化或跨重启 checkpoint。
- 3D 舞台只消费协同状态，不决定业务成功。
